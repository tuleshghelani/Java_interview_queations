# Java Interview Questions (Advanced Level)

## 101. What is Lock Elision and Lock Coarsening in the JVM?
Lock Elision and Lock Coarsening are advanced Just-In-Time (JIT) compiler optimizations in the JVM designed to minimize the overhead of synchronization in multithreaded applications.

### Lock Elision
Lock Elision is an optimization where the JIT compiler removes synchronization from a block of code entirely if it can prove that the lock object is never shared between multiple threads. 
This is achieved through **Escape Analysis**. If the compiler determines that the object being locked does not "escape" the current thread (meaning no other thread can possibly obtain a reference to it), it knows that there can be no contention. 

**Example:**
```java
public void appendStrings(String s1, String s2) {
    // StringBuffer is thread-safe and its methods are synchronized
    StringBuffer sb = new StringBuffer(); 
    sb.append(s1); // JIT removes the lock acquisition here
    sb.append(s2); // JIT removes the lock acquisition here
    System.out.println(sb.toString());
}
```
In this example, the `StringBuffer` object is created locally and never leaves the `appendStrings` method. The JIT compiler elides (removes) the locks from the `append` operations, making it perform exactly like an unsynchronized `StringBuilder`.

### Lock Coarsening
Lock Coarsening (or Lock Merging) is an optimization where the JIT compiler combines multiple adjacent or nested synchronization blocks that use the **same lock** into a single, larger synchronization block.
Acquiring and releasing a lock has overhead. If a thread repeatedly acquires and releases the same lock in a loop or in consecutive statements, the JVM optimizes this by expanding the lock's scope to cover the entire sequence, doing only one lock acquisition and one release.

**Example:**
```java
public void updateData(Object lock) {
    synchronized(lock) {
        doStep1();
    }
    synchronized(lock) {
        doStep2();
    }
    synchronized(lock) {
        doStep3();
    }
}
```
The JIT compiler will optimize the above code into a single block:
```java
public void updateData(Object lock) {
    synchronized(lock) {
        doStep1();
        doStep2();
        doStep3();
    }
}
```

### Why are these important?
Both optimizations allow developers to write clean, thread-safe code (often utilizing thread-safe standard library classes) without paying a severe performance penalty when synchronization is logically redundant or poorly scoped. However, it is still a best practice for developers to use appropriate concurrent data structures and minimize lock scopes themselves, as JIT optimizations are heuristic-based and not guaranteed for complex code paths.

## 102. What is Class Data Sharing (CDS) and Application Class Data Sharing (AppCDS) in the JVM?
Class Data Sharing (CDS) and Application Class Data Sharing (AppCDS) are JVM features designed to reduce the startup time and memory footprint of Java applications.

### How Class Loading Normally Works
When the JVM starts, it loads classes from the disk (JAR files or directories), parses their `.class` files into internal data structures, and stores them in the Metaspace. This process involves verification, resolving constants, and building metadata, which consumes CPU cycles and increases startup time. Furthermore, if you run multiple JVM instances on the same machine, each JVM independently loads and parses the exact same standard library and framework classes, wasting physical memory.

### Class Data Sharing (CDS)
Introduced in earlier Java versions, CDS aims to optimize this process. It works by taking the parsed, internal JVM representation of loaded classes and dumping them into a memory-mapped archive file (usually named `classes.jsa` — Java Shared Archive) on disk.
When a new JVM instance starts, instead of parsing those `.class` files from scratch, it simply maps the shared archive directly into its memory space. 
- **Startup Time Improvement:** Directly mapping memory is significantly faster than reading, parsing, and verifying individual `.class` files.
- **Memory Footprint Reduction:** If multiple JVM processes run on the same physical machine, the OS can share the memory-mapped archive across all of them. The read-only parts of the class metadata are shared in physical RAM, vastly reducing the overall memory footprint.

### Application Class Data Sharing (AppCDS)
Originally, CDS only supported the core Java standard library classes (like `java.lang.String`). Application Class Data Sharing (AppCDS) extends this feature to allow developers to include their own application classes and third-party library dependencies (like Spring Framework or Hibernate classes) in the shared archive. 

AppCDS became commercially available in Java 10 and was fully open-sourced and made a standard feature in Java 13 with the introduction of Dynamic CDS Archives.

### Dynamic CDS Archives (Java 13+)
In older versions, creating an AppCDS archive was a tedious multi-step process: you had to run the application to record a list of loaded classes, and then run it again to dump the archive.
With Dynamic CDS Archives, the process is dramatically simplified. You can instruct the JVM to automatically create the archive upon application exit:
```bash
java -XX:ArchiveClassesAtExit=my-app.jsa -cp my-app.jar com.example.Main
```
And then use the generated archive on subsequent runs:
```bash
java -XX:SharedArchiveFile=my-app.jsa -cp my-app.jar com.example.Main
```

### Why are CDS and AppCDS important?
In modern cloud-native and microservices architectures, where developers often deploy hundreds of identical JVM instances across a Kubernetes cluster or rely on serverless functions (where fast startup, or "cold start" reduction, is critical), CDS and AppCDS provide immediate, "free" performance benefits. They drastically cut down JVM warmup time and lower memory costs without requiring any changes to the application code.

## 103. What is Tiered Compilation in the JVM, and how do the C1 and C2 compilers work together?
Tiered Compilation is a JVM feature (enabled by default since Java 8) that combines the fast startup times of client-side compilation with the high peak performance of server-side compilation by using multiple compilation tiers.

### The Problem: Startup vs. Peak Performance
Historically, the HotSpot JVM came with two distinct Just-In-Time (JIT) compilers:
1. **Client Compiler (C1):** Optimizes code quickly but performs simple, local optimizations. It starts up very fast and uses less memory, but the resulting machine code is only moderately optimized.
2. **Server Compiler (C2):** Performs highly sophisticated, global optimizations (like escape analysis, loop unrolling, and inlining). It takes significant time and CPU resources to compile code, but it produces highly optimized machine code for peak throughput.

Before Tiered Compilation, developers had to choose at JVM startup between `-client` (fast startup, lower peak performance) and `-server` (slow startup, high peak performance).

### Tiered Compilation Levels (0 to 4)
Tiered Compilation solves this by dividing execution into five distinct levels (or tiers) and dynamically transitioning methods between them:

- **Level 0 (Interpreted Code):** The JVM starts by running all bytecodes through the Interpreter. This requires zero compilation time. The interpreter tracks how often a method is called (invocation counters) and how often loops backedge (backedge counters).
- **Level 1 (Simple C1 / No Profiling):** If a method becomes "hot" but is very simple/trivial, the JVM compiles it using the C1 compiler with no profiling instrumentation. The compiled code is simple and executes quickly.
- **Level 2 (Limited C1 / Basic Profiling):** The JVM compiles the method using the C1 compiler with light profiling. This level is typically used when the Level 3 compilation queue is full, to get compiled code running quickly while waiting.
- **Level 3 (Full C1 / Full Profiling):** The JVM compiles the method using the C1 compiler and inserts profiling code (to track branch probabilities, type feedback, etc.). This is the standard intermediate step before C2 compilation.
- **Level 4 (C2 Compiler):** Once a method gets extremely hot and has gathered enough profiling data at Level 3, it is compiled by the C2 compiler using that profiling data. This produces the most optimized native code.

### Typical Compilation Lifecycle
A method typically moves through the following stages:
1. **Level 0 (Interpreter)** -> **Level 3 (Full C1)** -> **Level 4 (C2)**
2. If the C2 queue is busy, the method might stay at Level 3 longer.
3. If the C1 queue is overloaded, a method might go directly from Level 0 to Level 4 (though without C1 profiling, this is less common).
4. **Deoptimization:** If the C2 compiler optimizes code based on profiling assumptions that are later invalidated (e.g., a class loader loads a new subclass, breaking a monomorphic inline cache), the compiled code is discarded. Execution falls back to **Level 0 (Interpreted)**, and the profiling/compilation cycle starts over.

### Diagnostic and Tuning Flags
- `-XX:+TieredCompilation`: Enables tiered compilation (default in Java 8+).
- `-XX:-TieredCompilation`: Disables tiered compilation, defaulting to C2 only (since `-server` is default).
- `-XX:TieredStopAtLevel=N`: Stops compilation at a specific level (e.g., `-XX:TieredStopAtLevel=1` forces the JVM to only use Level 1, bypassing C2 entirely, which is useful for short-running container applications or serverless functions to minimize CPU spikes during startup).

### Why is this important?
Tiered Compilation provides the best of both worlds: JVM applications start up and warm up quickly using C1, while eventually reaching the same extreme peak throughput as C2. This is crucial for modern containerized microservices, where fast startup/health checks and high peak performance are both required.
